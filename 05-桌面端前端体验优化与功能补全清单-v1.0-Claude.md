# AgentFlow 桌面端前端体验优化与功能补全清单 v1.0

> 交付对象：Claude（桌面端实现与视觉优化）
> 仓库：AgentDock
> 适用范围：`apps/desktop` 及必要的 Tauri 查询接口
> 关联文档：`01-核心实现方案-v1.1-Codex.md`、`02-桌面端UI实现方案-v1.1-Claude.md`、`03-接口契约与验收清单-v1.1.md`
> 文档目的：记录“当前哪里不好、应该怎么改、如何判断改好了”，以后任何人修改前端时都能逐项对照。

## 1. 本轮目标

这次不是简单换颜色、加阴影或放大图标，而是重新整理桌面端的信息层级，让用户在 3 秒内回答：

1. 任务现在有没有在运行？
2. 当前由哪个 Agent 做什么？
3. 是否需要用户处理？
4. 本轮真正完成了什么、还差什么？
5. 预算、安全、质量和交付是否正常？

前端必须把后端已经实现的治理能力变成用户看得懂、能操作、能确认的产品功能，不能只在日志或设置深处出现。

### 1.1 状态标记

`✅` 已有基础；`🟡` 后端已有但前端不完整；`🔴` 后端已有但前端无有效入口；`⚪` 功能可用但体验需要优化。

### 1.2 不在本轮范围内

- 不改 Orchestrator 的核心状态机语义。
- 不手写或直接修改自动生成绑定；类型变化应由生成流程产生。
- 不为了视觉效果隐藏真实失败、预算未知、恢复失败或安全风险。
- 不把技术日志删掉；技术日志保留在“技术详情”，只是不默认打扰普通用户。

## 2. 当前页面的核心问题

| ID | 截图与代码反映的问题 | 直接影响 |
|---|---|---|
| P0-01 | 圆点、用户、系统、Agent、阶段和结果处于同一层；revision、run、fallback 没有父子关系 | 看不懂任务如何推进，也不知道是否仍在运行 |
| P0-02 | 普通日志仍可能出现 JSON、命令、临时路径和协议事件 | 用户找不到真正的改动、问题和结论 |
| P0-03 | 中间 run 列表平铺；图标小；角色、Provider、结果挤在一行 | 无法区分主 Agent、降级、验证器和委员会成员 |
| P0-04 | 时间线、run 列表、日志重复信息却没有主从联动 | 信息很多但选择上下文不一致 |
| P0-05 | 预算、质量、复现、CI、回滚、队列和恢复状态藏得过深 | 用户必须到处切页才能确认任务是否健康 |

`scheduler slot`、heartbeat、result 文件等内部事件不应占据主时间线；失败和降级必须挂在原 run 下；普通用户默认只看有效文字、改动、验证、问题与下一步。

## 3. P0：重构任务执行树

### 3.1 用“轮次 → 阶段 → run”替换平铺时间线

建议结构：

```text
▼ r3 当前轮                                  进行中  12 分 08 秒
  ✓ 计划      Claude Code                    已批准
  ✓ 开发      Codex                          +8 文件  3 分 21 秒
  ✓ 验证      本机                           12/12 通过
  ▼ 审查委员会                               2/3 完成
    ✓ Claude Code                            通过
    ! DeepSeek API                           发现 2 个问题
    ↻ OpenAI API                             额度不足 → 已降级
      … Kimi API                             审查中  00:18
  ○ 人工批准                                 尚未开始

▶ r2 已结束                                  要求返工  18 分钟
▶ r1 已结束                                  要求返工  14 分钟

系统详情（默认折叠）                         8 条
```

层级规则：

- 第一层是 revision，例如 `r3 当前轮`。
- 第二层是业务阶段：计划、开发、验证、审查、人工批准、交付。
- 第三层才是具体 run 或审查委员会成员。
- Provider fallback 必须作为原 run 的子节点展示，不能成为独立顶层事件。
- scheduler、heartbeat、result file、checkpoint 等内部事件统一放进“系统详情”。
- 默认展开当前 revision 和当前阶段；过去 revision 默认折叠成一行总结。
- 点击阶段展示阶段总结；点击 run 展示该 run 的有效文字；点击技术详情才看底层事件。

### 3.2 圆圈、Agent 与状态必须分工

不要再让一个小圆圈同时表达角色和状态。建议使用三条独立视觉通道：

| 信息 | 表达方式 | 示例 |
|---|---|---|
| 谁在执行 | 28–32 px 官方 Provider 图标 + 名称 | Codex、Claude Code、DeepSeek |
| 在做什么 | 阶段图标 + 中文阶段名 | 开发、验证、审查、交付 |
| 执行结果 | 状态图标 + 文字 + 边框色 | 进行中、通过、失败、等待你 |

节点规范：

- 用户节点：人物/“你”标识，琥珀色，仅用于需要人工决策。
- Agent 节点：官方 Provider 图标，图标不小于 28 px。
- 验证节点：盾牌或检查图标，不伪装成某个 AI Provider。
- 调度/系统节点：16–20 px 中性图标，仅在系统详情出现。
- 成功：勾 + “通过/完成”；失败：叉 + “失败”；运行：旋转或柔和脉冲 + “进行中”。
- 颜色不是唯一信号，必须同时有图标或文字。
- 当前节点可以使用轻微脉冲；遵守 `prefers-reduced-motion`。

### 3.3 只保留对用户有意义的事件

主执行树允许出现：

- 计划生成、计划待批准、计划已批准或被驳回；
- 开发开始、完成、失败或被中断；
- 验证开始、结果摘要；
- 审查开始、各成员结论、委员会最终结论；
- 返工原因和新一轮开始；
- 等待用户、用户批准/驳回/补充指引；
- PR/MR、CI、合并、回滚结果；
- 预算阻断、远程节点不可用、恢复接管失败等需要处理的异常。

以下内容默认隐藏到系统详情：

- `scheduler slot`；
- heartbeat、轮询和内部状态同步；
- result 文件移动、读取和校验过程；
- 原始 `provider fallback` 事件名；
- command execution、tool call 参数；
- 临时目录、内部 ID、IPC 和 daemon 细节。

### 3.4 当前运行状态要持续可见

当前 revision 顶部必须有一条持续状态：

- `Codex 正在开发 · 已运行 02:41`；
- `正在本机验证 · 8/12 已完成`；
- `审查委员会 · 2/3 已返回`；
- `等待 Provider 限流恢复 · 预计 14:35 后重试`；
- `后台服务已重连 · 正在接管仍运行的 Agent`；
- `排队第 2 位 · 前面 1 个任务`。

若 10 秒以上没有新日志，但进程仍存活，应显示“仍在运行，最近活动于 10 秒前”，不能表现为静止或卡死。

### 3.5 执行树验收标准

- [ ] 用户不展开系统详情时，看不到 `scheduler slot`、原始事件名和重复 fallback。
- [ ] 当前 revision、阶段、run 三层关系一眼可辨。
- [ ] 多 Agent 审查委员会显示为同一审查阶段下的并行分支。
- [ ] Provider 降级显示在原 run 下，并说明“为什么降级”和“降级到谁”。
- [ ] 当前运行步骤有文字、动态图标、已用时和最近活动时间。
- [ ] 过去 revision 默认折叠，折叠行包含结论、耗时和改动摘要。
- [ ] 点击树节点会同步中间列表和右侧详情，不出现三处选择不一致。
- [ ] 仅靠黑白截图也能区分进行中、成功、失败、等待人工和未开始。

## 4. P0：重做日志信息架构

### 4.1 默认只展示“结果优先”的内容

右侧默认视图改成三个层次：

1. **本次结果**：一句话结论。
2. **主要内容**：做了什么、改了什么、测试结果、发现的问题、下一步。
3. **实时进展**：运行中才显示，使用人类可读的短句。

建议结果卡结构：

```text
开发完成

完成了什么
• 新增待办事项的 add/list/done/remove 命令
• 使用临时文件 + rename 保证原子写入

验证
• 12 项测试全部通过
• typecheck 通过

仍需注意
• Windows rename 行为还没有真实环境验证
```

### 4.2 内容过滤规则

普通视图只允许：

- Agent 的自然语言消息；
- 结构化结果中的 summary、changes、tests、issues、next action；
- 审查结论和可执行建议；
- 用户真正需要处理的错误说明。

普通视图不得出现：

- JSON / JSONL 原文；
- `item.completed`、`command_execution` 等协议字段；
- shell 命令全文和工具参数；
- 临时文件绝对路径；
- schema、task UUID、内部 result 文件路径；
- 将 JSON 字符串再次转义后的大段文本。

解析失败时：

- 先递归解析嵌套 JSON 字符串；
- 从 `agent_message.text`、`aggregated_output.summary`、`result.summary` 等候选中提取文字；
- 若仍失败，普通视图显示“未能提取可读总结”，并提供“查看技术详情”；
- 禁止把原始 JSON 作为降级结果塞回主要内容。

### 4.3 技术详情保留但后置

技术详情使用抽屉或独立子标签，包含：

- 系统事件；
- 工具调用；
- 命令与退出码；
- 原始协议；
- stdout/stderr；
- 搜索、复制、下载脱敏日志；
- 显示/隐藏时间戳；
- 自动跟随最新日志开关。

技术详情默认关闭，并记住用户偏好，但新用户永远从主要内容开始。

### 4.4 日志验收标准

- [ ] Codex、Claude、Gemini、Qwen、API Provider 和历史日志都经过同一可读内容策略。
- [ ] “主要内容”中不存在 `{\"type\":...}`、shell 命令或临时目录。
- [ ] 解析异常不会白屏，也不会把原始 JSON 泄漏到普通视图。
- [ ] 最终总结优先显示结构化结果，不依赖最后一行日志碰运气。
- [ ] 运行中至少每 10 秒刷新“最近活动”或明确显示仍在运行。
- [ ] 10k 行技术日志仍使用虚拟列表，切换标签不掉帧。
- [ ] 用户能复制文字总结，也能单独导出经过脱敏的技术日志。

## 5. P0：任务详情页整体布局

### 5.1 推荐布局职责

| 区域 | 唯一职责 | 不应承担的内容 |
|---|---|---|
| 顶部任务头 | 总状态、当前行动、关键风险、Agent 分工 | 完整日志和所有技术字段 |
| 左侧执行树 | 在轮次和阶段之间导航 | 逐条底层系统事件 |
| 中间 run/成员列表 | 选择某阶段的具体执行者或尝试 | 重复完整时间线 |
| 右侧内容区 | 展示所选节点的结果和证据 | 再做一次全局导航 |
| 底部行动栏 | 当前最重要的一组用户操作 | 永久堆放无关按钮 |

中间列表在只有一个 run 时可以自动收起，为右侧内容留出空间；审查委员会或多次降级时再展开。

### 5.2 顶部任务头增强

顶部第一行保留：任务号、标题、总状态。

第二行改为可读摘要：

- `Codex 开发 → Claude Code 审查`，图标 28–32 px；
- `当前 r3 · 审查中 · 2/3 成员完成`；
- `本机验证`或`远程：Mac Studio`；
- `预算剩余约 42%`；
- `交付：GitHub PR #12 · CI 运行中`。

分支、base SHA、完整 commit 等次要信息放进“任务信息”popover，避免头部成为一排灰色技术文字。

### 5.3 Run 列表重构

- 按当前阶段分组，不再把所有 revision 的所有 run 一次平铺。
- 默认只显示当前 revision；历史轮次从执行树进入。
- 每行最少 48–56 px，高度足以容纳 28 px 图标、名称和一行结果。
- 第一行：角色 + Provider 名称 + 状态文字。
- 第二行：耗时 + 简短结果/失败原因。
- 降级尝试显示 `首选`、`降级 1`、`降级 2` 标签。
- 审查委员会显示 `成员 1/3` 与其投票结论。
- 当前 run 有选择边框和进行中标识，不能只靠 6 px 小圆点。

### 5.4 图标尺寸规范

| 场景 | 建议尺寸 |
|---|---:|
| 设置页 Provider 主列表 | 40 px |
| 任务头部 Agent 分工 | 30–32 px |
| 执行树 Agent 节点 | 28–30 px |
| Run 列表 | 28–32 px |
| 紧凑标签/委员会来源 | 22–24 px |

- OpenAI/Codex 官方 SVG 的图形本体需要填满 tile 的约 70%–78%，不能因为 SVG viewBox 留白而显得比其他图标小。
- 所有官方图标必须保持原始比例，不拉伸、不自行重绘、不用近似品牌图形。
- 图标必须同时配文字或 tooltip，不能只靠图形识别。

## 6. 已实现核心能力与前端缺口清单

### 6.1 编码前计划审批 — `🟡 部分实现`

已有：计划弹窗、步骤、风险、允许路径、批准和驳回。

需要补全：

- [ ] 在执行树中形成独立“计划”阶段，而不是只靠底部行动栏。
- [ ] 允许路径改成可折叠目录树，突出通配范围和高风险路径。
- [ ] 显示计划版本变化：v1 被驳回后，v2 哪些步骤、风险、路径发生变化。
- [ ] 将 plan seal/hash 翻译为“批准后计划已锁定”；技术 hash 放详情。
- [ ] 若编码范围越界导致重新审批，明确展示越界文件和旧计划/新计划差异。
- [ ] 批准前摘要显示预计阶段、风险数量、允许修改文件范围，而不是只有长文本。

### 6.2 Token、费用和时间硬预算 — `🟡 部分实现，存在显示错误风险`

已有：任务创建时配置预算、治理页进度卡、超限后调整预算并继续。

需要补全：

- [ ] 不能把 `costUsd === null` 显示成 `$0`；应显示“未知”并说明未知 run 数。
- [ ] 使用 `tokensKnown`、`costKnown`、`unknownTokenRuns`、`unknownCostRuns`。
- [ ] 显示已预留但尚未结算的 `tokensReserved`、`costReservedUsd`。
- [ ] 区分 `hard / soft / unavailable`，文案为“硬限制 / 仅提醒 / Provider 不提供用量”。
- [ ] 进度条区分已结算、已预留和剩余，不只画一段 used。
- [ ] 70%、90%、100% 有分级提醒，只有 100% 才进入明确阻断。
- [ ] 任务头部提供一行预算健康摘要，详情仍留在治理页。
- [ ] 调整预算弹窗在用量未知时不能用 0 自动建议新上限。
- [ ] 显示“提高预算后从哪个安全阶段继续”，避免用户以为会从头重跑。

### 6.3 可复现运行与质量评估 — `🟡 部分实现`

已有：质量分、检查项、固定 commit 复验按钮、基础 manifest hash。

需要补全：

- [ ] 显示复现等级：固定提交、环境锁定、完全密封，并解释每级含义。
- [ ] 展示工具版本、系统依赖、环境摘要、submodule、LFS、容器 digest 和外部依赖。
- [ ] 明确展示 `limitations`，不能只展示几个 hash 就暗示完全可复现。
- [ ] hash 默认缩略，提供复制和“查看完整清单”，不占据主阅读路径。
- [ ] 复验时显示进行中、当前检查、耗时、执行位置和完成结果。
- [ ] 比较原验证与复验结果：一致 / 不一致，以及差异在哪里。
- [ ] 质量分旁边解释扣分来源和是否为硬门禁，不把分数做成无意义仪表盘。
- [ ] 复验失败提供下一步：查看日志、检查环境漂移、重试或更换节点。

### 6.4 合并后的撤销与回滚 — `🟡 部分实现`

已有：undo/revert 两种操作和确认对话框。

需要补全：

- [ ] 用普通语言区分“撤销刚完成的本地合并”和“创建一个反向提交”。
- [ ] 执行前预览目标分支、merge commit、预计影响文件和是否存在后续提交。
- [ ] 不满足 undo 条件时直接禁用，并说明必须使用 revert 的具体原因。
- [ ] 执行中显示阶段和进度；完成后展示新 HEAD/revert commit。
- [ ] 回滚结果写入执行树“交付”阶段，并提供打开 Diff。
- [ ] 回滚失败提供可恢复说明，不能只弹一条 toast。

### 6.5 GitHub/GitLab PR 与 CI — `🟡 部分实现`

已有：交付模式、PR/MR URL、CI 总状态、刷新操作和合并门禁。

需要补全：

- [ ] 显示每一个 CI check，而不仅是 `pending/passed/failed` 总词。
- [ ] 展示 PR/MR 的 draft、open、merged、closed 状态，以及 base/head 分支。
- [ ] CI 失败显示失败 check、简短原因和“打开远端日志”。
- [ ] 明确说明“CI 未通过，因此未合并”，避免用户误以为任务卡死。
- [ ] 交付状态进入顶部摘要和执行树，不只在治理页。
- [ ] 自动刷新与手动刷新要有最近刷新时间。
- [ ] 检测远端 HEAD/base 变化或目标分支前进时，显示重新检查结果和下一步。

### 6.6 远程执行节点 — `🟡 部分实现`

已有：节点增删改、SSH 检查、在线状态，任务可选择验证节点。

需要补全：

- [ ] 节点卡显示最后检查时间、延迟、平台、Git 和关键工具版本。
- [ ] 连接检查显示分步结果：DNS/SSH/工作目录/Git/验证命令，而不只给在线或失败。
- [ ] 任务执行树和验证结果都标明具体节点。
- [ ] 远程运行显示上传归档、执行、回传报告、清理临时目录四个阶段。
- [ ] 节点离线时提供“改用本机”“选择其他节点”“稍后重试”。
- [ ] 编辑节点时解释不保存密码/私钥，使用系统 SSH 配置。
- [ ] 历史任务引用的节点删除失败时，直接引导“停用”而不是只报错。

### 6.7 常驻 daemon、调度和资源治理 — `🔴 任务级操作入口缺失`

## 10. Claude 实施约束

- 单个手写 TS/TSX 文件尽量不超过 600 行；超过时按职责拆分。
- 执行树至少拆分为：数据归一化、树模型、节点组件、系统详情四部分。
- 日志解析和 UI 渲染分离；Provider 兼容规则放纯函数并写测试。
- 复用 `ProviderIcon`、`AgentMark`、`StateBadge`，不要在不同页面复制品牌和状态映射。
- 后端事件名不得直接成为用户文案，统一经过 copy/adapter 层。
- 不直接修改 `src/generated/bindings.ts`；绑定必须通过项目生成命令更新。
- 新增交互必须同步开发 shim，覆盖成功、失败、运行中、未知、空状态。
- 不使用未经确认的非官方品牌图标。
- 不删除技术日志能力，只改变默认信息层级。
- 不用轮询制造高频闪烁；优先订阅事件，轮询只作恢复兜底。

执行树至少拆成数据归一化、树模型、节点组件、系统详情；日志至少拆成内容归一化、结果摘要、实时进展、技术日志；治理页按预算、复现质量、交付回滚拆分。目录名称可调整，但解析/模型必须与展示解耦。

## 13. 最终验收定义

本轮前端优化完成，不是指页面更漂亮，而是同时满足：

1. 新用户不看技术日志，也能理解一个任务从计划到交付发生了什么。
2. 当前正在运行、排队、等待用户、失败或恢复接管都不会产生歧义。
3. Agent 身份、业务阶段和执行状态使用不同视觉元素表达。
4. 普通日志不再出现 JSON、命令和协议噪声。
5. 后端已实现的预算、可复现、CI、回滚、调度、恢复、安全和 Git 兼容能力都有可发现的前端入口。
6. 危险操作可预览影响、可取消、失败后可恢复。
7. 所有页面在常见窗口尺寸、200% 缩放、键盘操作和 reduced motion 下可用。
8. 每项功能在第 12 节留下改动文件、验证证据和前后截图，方便以后继续维护。
