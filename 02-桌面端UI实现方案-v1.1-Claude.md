## 1. 产品立意与设计定位

**主体**：一个"本地调度塔"。用户是独立开发者，同时是这条流水线上唯一的人类权威：两个 AI（Claude Code 开发、Codex 审查）在他名下的仓库里干活，他负责放行。

**界面的唯一职责**：让用户在任何时刻 3 秒内回答两个问题——"现在需要我吗？""需要我做什么？"其余一切（日志、diff、时间线）都是为这个判断服务的证据。

**设计基调**：安静的机器 + 醒目的人。整个界面是低饱和、冷灰蓝的仪表环境；**琥珀色是全应用唯一保留给"需要人类介入"的颜色**——审批栏、BLOCKED 状态、合并冲突、控制面文件警告。用户被训练成"看到琥珀就是在叫我"。这是本设计的签名元素，任何其他装饰不得使用琥珀。

明确排除的俗套：不做米白底+衬线大标题+陶土点缀，不做纯黑底+荧光绿，不做报纸细线风。这是一台工作中的仪器，不是营销页。

全部以 CSS 变量落在 `src/styles/tokens.css`，组件只允许引用 token，不允许裸写色值。

### 2.2 字体

- UI 正文：`"IBM Plex Sans", "IBM Plex Sans SC", system-ui` —— 工程仪器气质，且有官方简中字重，中英混排不塌。
- 代码/日志/ID/SHA：`"IBM Plex Mono", ui-monospace`。任务号 `TASK-007`、SHA 短码、分支名一律 Mono。
- 字号阶梯：12（元数据）/ 13（正文密排）/ 14（正文）/ 16（区块标题）/ 20（页标题）。桌面工具，整体偏紧凑，行高 1.5。
- 字体文件本地打包（`@fontsource/ibm-plex-sans` 等），**禁止运行时拉外网字体**（桌面应用可能离线）。

### 2.3 空间与形态

- 间距基数 4px；面板内边距 16；列表行高 36（紧凑）。
- 圆角：控件 6px，面板 8px；不做大圆角。
- 边框用 `--line` 发丝线；阴影只给悬浮层（menu/dialog）一档。
- 动效：120–160ms ease-out，只用于面板展开、状态徽章切换、审批栏滑入；`prefers-reduced-motion` 全部退化为即时切换。日志滚动与进度不加动画。

## 3. 信息架构与路由

```text
/                       → 重定向：无项目→/onboarding，有→/p/<projectId>
/onboarding             → 环境检测 + 导入第一个项目
/p/:projectId           → 任务列表（该项目）
/p/:projectId/t/:taskId → 任务详情（核心页面）
/settings               → 全局设置（含各项目设置入口）
```

布局骨架：左侧 220px 项目栏（可折叠到 56px 图标态）+ 主区。项目栏底部固定"环境状态"胶囊（onboarding-check 结果：daemon、Git、四种 CLI 与 OpenAI/Anthropic/DeepSeek API 的圆点，阻塞项出现时整胶囊转琥珀，点击进设置）。

## 4. 页面规格

### 4.1 Onboarding / 环境检测

目的：首启把三件事办完——检测 CLI、导入 Git 项目、（可选）确认默认 Agent 分工。

- 检测卡片（Git / Claude Code / Codex / Gemini CLI / Qwen Code / OpenAI API / Anthropic API / DeepSeek API）：CLI 显示路径、版本和可查询登录状态；API 显示协议、base URL、模型和凭据是否可用但绝不显示密钥；未找到的 Gemini/Qwen 显示"适配器已就绪，可稍后安装"而不是阻断。手填 CLI 路径走 `commands.envSetCliPath`。页面总览数据优先使用 `commands.onboardingCheck()`，完成引导后调用 `commands.onboardingComplete()`。
- 导入项目：目录选择（Tauri dialog）→ `commands.projectImport(path)`；错误 `PROJECT_NOT_GIT` 给明确指引。
- 空态文案见 §8。检测中给骨架，不给转圈遮罩。

### 4.2 任务列表页

```text
┌────────────┬──────────────────────────────────────────────┐
│ 项目栏      │  <项目名>  分支 main          [新建任务]      │
│  ▸ proj A  │  ── 需要你（琥珀分组，置顶，可为空） ──        │
│  ▸ proj B  │  ⚠ TASK-012 等你批准        r2 · 2 分钟前    │
│            │  ⚠ TASK-009 审查要求返工已达上限  · 1 小时前  │
│  [env ●●●] │  ── 进行中 ──                                │
│            │  ◍ TASK-013 开发中 · Claude   r1 · 刚刚      │
│            │  ── 其他 ──（已合并/已取消 默认折叠）          │
└────────────┴──────────────────────────────────────────────┘
```

- 分组规则是产品逻辑不是排序偏好：`WAITING_FOR_HUMAN_APPROVAL / BLOCKED / MERGE_CONFLICT / APPROVED(待合并)` 进"需要你"组；进行态进"进行中"；`MERGED/CANCELLED` 折叠到"已完结"。
- 行内容：状态徽章（§6 StateBadge）、`TASK-<seq>`（Mono）、标题、当前 revision、相对时间、执行中任务的 Agent 小徽标。
- 数据：`commands.taskList(projectId)` + 订阅 `task:changed` 事件做行级更新（整表 invalidate 仅在事件缺口兜底轮询 30s 一次）。

### 4.3 新建任务（对话框）

字段：标题、描述（多行，说明会原文交给开发 Agent）、开发 Agent / 审查 Agent（默认 Claude Code / Codex，可换但**不允许两者相同**，前端校验+文案说明"同源审查会显著降低缺陷检出，本工具的核心价值就是异构互审"）、目标分支（默认项目 default_branch）、最大返工轮数（默认 3）。提交 → `commands.taskCreate` → 跳详情页；"创建并立即开始"副按钮再调 `taskStart`。

### 4.4 任务详情页（核心页面）

```text
┌──────────────────────────────────────────────────────────────┐
│ TASK-012  修复登录空指针        [状态徽章]   r1 r2(当前)      │
│ 分支 agentflow/TASK-012 · base a1b2c3d · 开发 Claude·审 Codex│
├───────────┬──────────────────────────────────────────────────┤
│ 时间线     │   [概览] [日志] [Diff] [审查]                    │
│ ● 创建     │                                                  │
│ ● 开发 r1  │   （右侧为当前标签内容区）                        │
│ ● 验证 ✕   │                                                  │
│ ● 返工 r2  │                                                  │
│ ● 审查 ✓   │                                                  │
│ ◉ 等你批准 │                                                  │
├───────────┴──────────────────────────────────────────────────┤
│ ⚠ 审批栏（仅"需要你"状态出现，琥珀，吸底）                     │
│  Codex 审查通过 · commit 9f8e7d6 · +214 −38 · [驳回] [批准]  │
└──────────────────────────────────────────────────────────────┘
```

**时间线（左栏，签名结构件）**：由 `commands.eventsList(taskId)` 渲染的垂直调度日志；每节点=一次状态迁移或 run 起止，含 actor 徽标（orchestrator 齿轮 / Claude / Codex / 人类）。它同时是 revision 导航：点节点切换右侧上下文到对应 revision。

**概览**：任务描述、开发者每轮 result summary（此处允许展示，审查输入里没有是刻意的——概览页加一行说明）、BLOCKED 时展示 reason 的人话解释 + 对应动作按钮（见下）。

**日志**：run 选择器（该任务全部 run，按时间倒序，validator 也在内）→ RunLogViewer（§6）。实时 run 订阅 `run:log` 事件；历史 run 用 `commands.runLogTail` 分段加载 `agent-events.jsonl` 归一化事件（不是原始 JSONL——渲染 AgentEvent：工具调用一行一条、助手文本折叠块、result 摘要）。

**Diff**：`commands.diffGet(taskId, revision)` → 左侧文件树（含 numstat），右侧 Monaco DiffEditor **read-only**；`flagged` 文件在树中琥珀标记，选中时顶部横幅："该文件属于规则/控制面文件，Agent 修改了它——请确认这是你想要的"。`truncated=true` 时明示"diff 过大已省略 X 个文件的内容"。二进制文件只显示占位说明。

**审查**：按 severity 分组（critical/high → `--st-bad`，medium/low 中性）的 IssueCard 列表；点 file:line 跳转 Diff 标签对应位置；已 resolved 的置灰。头部是 decision 徽章 + 审查 summary。

**审批栏**：状态驱动的吸底琥珀栏，是全应用唯一的琥珀大面积使用点。
- WAITING_FOR_HUMAN_APPROVAL：`[驳回…] [批准]`。批准 → 二次确认对话框（展示 commit 短 SHA、±行数、flagged 警告若有）→ `commands.taskApprove(taskId, revision, commitSha, diffSha256)`；收到 `DIFF_STALE` → 提示"内容已变化，已为你刷新"并重载，**绝不静默重试**。驳回必填理由 → `taskReject`。
- APPROVED：`[我自己合并（标记完成）] [立即合并]`；`MERGE_PRECONDITION_FAILED` 按 detail 给出人话（"主仓库有未提交改动，请先处理"）。
- MERGE_CONFLICT：说明 + `[重试合并] [已手动解决，标记完成]`。
- BLOCKED：按 reason 渲染动作：`needs_clarification`（展示 Agent 的问题 + 回答输入框 → `taskResumeWithGuidance`）、`max_revisions`（`[补充指引继续] [直接送审批] [取消任务]`）、`review_block` / `run_failed` / `no_changes` / `validation_infra` / `worktree_missing` 各配一句人话解释 + 合理动作组合（映射表放 `src/copy/blocked.ts`）。

### 4.5 设置页

全局：CLI 路径覆盖（带"重新检测"）、默认超时、并发 run 上限、worktree 根目录、通知开关，以及"存储与隐私"卡片。存储卡片调用 `storageReport/storageCleanup` 展示日志、任务运行数据、artifact、cache、回收站占用；提供日志清理、任务移入回收站、恢复、清空回收站，彻底删除必须二次确认。项目级（进入具体项目的设置抽屉）：默认 Agent 分工、Developer/Reviewer 降级顺序、最大返工轮数默认值、环境变量剔除列表、**完全放权模式**开关——开启需输入项目名确认，开启后该项目所有页面顶部常驻细琥珀警示条"此项目已关闭命令确认防护"。

## 5. 数据层

- **唯一通道**：`import { commands, events } from "@/generated/bindings"`。禁止手写 `invoke("...")` 字符串。
- 服务端状态：TanStack Query 包命令调用，query key 规范 `['tasks', projectId]`、`['task', taskId]`、`['diff', taskId, rev]`、`['review', taskId, rev]`、`['events', taskId]`；**失效由事件驱动**（`task:changed` → 精准 setQueryData 该行 + invalidate 详情），不开启窗口聚焦自动重取，兜底轮询仅任务列表 30s。
- 事件订阅：`useTauriEvent(name, handler)` 统一封装（自动清理、StrictMode 防重复）。
- 日志：`logStore`（Zustand）按 runId 维护环形缓冲（上限 5000 行，超出丢头部并显示"已省略更早输出，点击加载"→ runLogTail 取历史）。`run:log` 已由后端按 ≤10 批/秒聚合，前端**再** rAF 合帧写入，渲染用虚拟列表。
- UI 状态（面板折叠、当前标签、选中 run）：Zustand `uiStore`，不进 Query。
- 错误处理：所有命令错误走统一 `toAppError`（03 §5 错误码表 → 人话文案映射 `src/copy/errors.ts`）；后台错误经 `app:error` 事件 → toast。toast 不用于成功确认（状态徽章本身就是确认），只用于错误与"已复制"类瞬时反馈。

## 7. 性能与可靠性要求（验收会测）

- 10k 行日志滚动不掉帧（虚拟列表 + 合帧写入）；日志页切走时取消订阅渲染但缓冲继续。
- 200 文件 / 单文件 5k 行的 diff 可开（文件内容按需加载，树先渲染）。
- 任一命令失败界面可恢复：不出现白屏（路由级 ErrorBoundary）、不出现永久 loading（所有 Query 配 error 态 UI）。
- 审批相关操作**零乐观更新**：一律等命令返回再变 UI。
- 事件断流兜底：详情页 60s 未收到任何事件且存在进行态 run → 静默 refetch 一次。

## 8. 文案规范（中文）

原则:按用户能控制/识别的事物命名，不暴露实现（说"等你批准"，不说 WAITING_FOR_HUMAN_APPROVAL；说"命令确认防护"，不说 dangerously-skip-permissions）。动作按钮说结果："批准并进入合并"而不是"提交"。错误 = 发生了什么 + 怎么办，不道歉不卖萌。

状态徽章文案（权威表，`src/copy/status.ts`）：

| status | 文案 | 色 |
|---|---|---|
| DRAFT | 草稿 | idle |
| READY_FOR_DEVELOPMENT | 排队开发 | idle |
| DEVELOPING | 开发中 | run |
| VALIDATING | 验证中 | run |
| READY_FOR_REVIEW | 排队审查 | idle |
| REVIEWING | 审查中 | review |
| READY_FOR_REVISION | 排队返工 | idle |
| REVISING | 返工中 | run |
| WAITING_FOR_HUMAN_APPROVAL | 等你批准 | human |
| APPROVED | 已批准 · 待合并 | human |
| MERGING | 合并中 | run |
| MERGE_CONFLICT | 合并冲突 | human |
| MERGED | 已合并 | ok |
| BLOCKED | 需要你处理 | human |
| CANCELLED | 已取消 | idle |

示例空态（任务列表）："还没有任务。新建一个，写清楚你想要什么——描述会原文交给开发 Agent。"

## 9. 可访问性与快捷键

- 全键盘可达；焦点环 2px `--st-run`，不隐藏。
- 状态不只靠颜色：徽章始终带文字；flagged 文件除琥珀点外带 ⚠ 字形。
- 快捷键：`Cmd/Ctrl+N` 新建任务；详情页 `1-4` 切标签；`Cmd/Ctrl+Enter` 在审批确认对话框内=确认。快捷键在 tooltip 标注。
- 日志/diff 区域文本全部可选中复制；SHA、分支名、路径点击即复制（toast"已复制"）。

## 10. 实现顺序与自检

1. tokens + 骨架布局 + 路由 + bindings 接入冒烟（envCheck 真调通）；
2. 任务列表 + 新建 + 事件订阅链路；
3. 详情页：时间线→日志→审查→审批栏→Diff（依赖重的放最后）;
4. Onboarding + 设置；
5. 性能项（§7）与 03 §8.E 逐项自评，结果回填 03 表格。

**禁止事项**：不 mock 契约绕过后端；不修改 crates 与 src-tauri；不引入运行时外网依赖（字体/CDN）；不对审批做乐观更新；琥珀色不得用于"需要人类"以外的任何用途。
