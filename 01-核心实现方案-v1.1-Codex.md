## 1. 系统概述

本系统是一个本地桌面应用：**Claude Code 开发 → Codex 独立审查 → Claude Code 返工 → 人工批准 → 合并**。开发/审查角色可互换。

核心原则（不变）：
- 本地优先；默认驱动已登录 CLI（Claude Code / Codex / Gemini CLI / Qwen Code），也可由用户显式选择 OpenAI/Anthropic/DeepSeek API 作为只读审查 Provider；CLI 未安装时适配器保持禁用，不自动下载安装；
- Orchestrator 是唯一调度权威，Agent 只能提交成果，不能控制其他 Agent；
- Agent 进程短生命周期（启动→执行→退出），Orchestrator 常驻；
- 任务可恢复、过程可审计、组件可替换。

v1.1 新增原则：
- **不可变审查对象**：一切审查、审批、合并都针对 revision commit SHA；
- **文件先于信号**：成果文件是权威，任何通知/退出码只是"去读文件"的提示；
- **控制面隔离**：Agent 可写的目录里不存在任何能影响调度决策的输入文件的权威副本；
- **权限委托**：命令执行的安全边界由各 CLI 自身权限系统实现，Orchestrator 负责正确配置它们并记录配置。

## 3. 关键设计决策（v1.1 定稿）

1. **中心调度 + 短生命周期 run**（不采用 tap 的对等常驻模型；论文后期实际也退化为无头长跑 + 人工协调）。
2. **SQLite 是当前状态与事件的唯一权威**；文件目录存放大对象（日志、patch、JSON 原文），DB 存路径与哈希。
3. **revision commit 制**：每轮开发结束由 Orchestrator 在任务分支上创建 commit，后续一切引用 SHA。
4. **规则注入而非文件生成**：`.agentflow/rules/*.md` 的内容由 Orchestrator 内联进每次 run 的 input.md；**不生成/不修改** 仓库中的 `CLAUDE.md` / `AGENTS.md`（若仓库已有则由 CLI 自行读取，且它们的变更会在 diff 中被高亮标记）。
5. **第一版用系统 Git CLI**（`--porcelain`/`-z` 解析），不用 libgit2。
6. **MCP 不进核心依赖**，M4 以 stdio 形式补充。
7. **调度进程与 Agent 生命周期分离**：`agentflowd` 由 launchd 常驻，Agent CLI/API 请求仍为短生命周期；UI 关闭不终止队列任务。
8. **API 能力边界**：首版 API Provider 只做审查，不直接获得本地文件写权限；开发仍由沙箱化 CLI 完成。密钥只从配置的环境变量或 macOS Keychain 读取，DB 不保存密钥。OpenAI 使用 Responses，Anthropic 使用 Messages，DeepSeek 使用 OpenAI-compatible Chat Completions。
9. **Provider 自动降级**：开发和审查分别维护可配置的 Provider 顺序；开发切换前将 AgentFlow 专属 worktree 重置到本轮起点，审查切换时排除实际生成该 revision 的 Provider。
10. **本地数据有生命周期**：本地优先不等于永久保存；原始日志、缓存和任务运行目录有独立保留策略，任务删除先进入可恢复回收站。
11. **Provider 边界版本化**：核心只依赖握手、健康检查、事件与结构化结果；厂商参数留在可独立更新的 stdio sidecar。协议降低核心改动频率，但上游发生破坏性变化时仍需更新对应 sidecar。

## 4. 目录与数据布局

### 4.1 应用数据目录（唯一运行时存储，Agent 不可达）

```text
<app_data>/                          # 由 tauri path API 解析，如 ~/Library/Application Support/com.agentflow.desktop
├── agentflow.db                     # 全局 SQLite（WAL）
├── logs/                            # tracing 滚动日志
├── cache/                           # 有容量上限的可再生缓存
├── .trash/                          # 删除任务的 7 天可恢复回收站
├── exports/                         # events 导出的 jsonl
├── providers/<package>/             # 可选外部 Provider 包（provider.json + 包内可执行文件）
└── projects/p<seq>/
    └── tasks/t<seq>/
        ├── runs/<run_id>/
        │   ├── input.md             # 本次 run 输入的权威副本
        │   ├── stdout.log           # 原始输出（Claude: JSONL；Codex: JSONL）
        │   ├── stderr.log
        │   ├── agent-events.jsonl   # 适配器归一化后的事件流（给 UI 回放）
        │   ├── result.json          # 开发 run 成果（从 worktree 收取后归档）
        │   └── last-message.json    # 审查 run 最终消息（codex -o 落盘）
        ├── reviews/r<N>/review.json
        └── artifacts/
            ├── r<N>.patch
            └── r<N>-tests.json
```

### 4.2 Worktree 布局（Agent 唯一可写区）

- 默认根：`<app_data>/wt/p<seq>/t<seq>/`（**刻意短路径**，规避 Windows MAX_PATH + node_modules 深嵌套；`worktree_root` 可在设置中改到与仓库同盘）。
- 每任务一个 worktree、一个分支 `agentflow/TASK-<seq>`；每个 revision 是该分支上的一个 commit（不为每个 revision 建新 worktree，避免依赖重装）。
- worktree 内约定两个目录，并**追加进 `<repo>/.git/info/exclude`**（对所有 worktree 生效、不进版本库，追加前去重）：
  - `.agentflow-in/`：Orchestrator 写入的本次输入（input.md、review-input.md、history.md、diff.patch）；每次 run 开始前整目录重建；
  - `.agentflow-out/`：Agent 写出成果（result.json）；run 结束后 Orchestrator 收取归档并清空。
- 由于被 exclude，`git add -A` 提交 revision 时天然不含这两个目录。

### 4.3 仓库内可提交配置（可选，人工维护）

```text
<repo>/.agentflow/
├── project.toml        # 见 03 文档 §6.1：validate 步骤(argv)、review 排除 globs、超时等
└── rules/
    ├── coding.md
    └── review.md
```

**危险开关（如跳过权限确认）绝不放在可提交配置里**，只存 DB 的 per-project settings。

## 5. 状态机（权威定义）

任务状态（`TaskStatus`）：

```text
DRAFT, READY_FOR_DEVELOPMENT, DEVELOPING, VALIDATING, READY_FOR_REVIEW,
REVIEWING, READY_FOR_REVISION, REVISING, WAITING_FOR_HUMAN_APPROVAL,
APPROVED, MERGING, MERGE_CONFLICT, MERGED, BLOCKED, CANCELLED
```

Run 状态（`RunStatus`）：`RUNNING, SUCCEEDED, FAILED, TIMED_OUT, CANCELLED, INTERRUPTED`。

任务级没有 FAILED/TIMED_OUT：run 失败一律进入 `BLOCKED(reason)`，人工从 BLOCKED 决定重试/给指引/取消。`BLOCKED` 必须携带 `blocked_reason`（见 03 §2 枚举）。

### 5.1 转移表（实现必须逐行单测）

| 当前状态 | 事件 | 目标状态 | 副作用 |
|---|---|---|---|
| DRAFT | user:start | READY_FOR_DEVELOPMENT | 记 base_commit=target HEAD；建 worktree+分支；写 info/exclude |
| READY_FOR_DEVELOPMENT | scheduler:slot | DEVELOPING | revision+=1；生成 input.md；spawn developer run |
| DEVELOPING / REVISING | run:succeeded ∧ result 有效 ∧ 有变更 | VALIDATING | `git add -A && commit` 生成 revision commit；写 task_revisions；生成 r<N>.patch |
| DEVELOPING / REVISING | run:succeeded ∧ 无变更 | BLOCKED(no_changes) | — |
| DEVELOPING / REVISING | result.status=needs_clarification | BLOCKED(needs_clarification) | question 存入 blocked detail |
| DEVELOPING / REVISING | run:failed/timed_out ∨ result 修复重试后仍无效 | BLOCKED(run_failed) | — |
| VALIDATING | scheduler:slot（无活动 run） | VALIDATING | spawn validator run（本状态可重入） |
| VALIDATING | 全部步骤通过 | READY_FOR_REVIEW | 写 r<N>-tests.json |
| VALIDATING | 有步骤失败 | READY_FOR_REVISION* | 测试报告进入下轮输入 |
| VALIDATING | 基础设施错误（spawn 失败等） | BLOCKED(validation_infra) | — |
| READY_FOR_REVIEW | scheduler:slot | REVIEWING | 生成 review-input.md + 精简 diff；spawn reviewer run |
| REVIEWING | decision=pass | WAITING_FOR_HUMAN_APPROVAL | — |
| REVIEWING | decision=request_changes | READY_FOR_REVISION* | issues 入库 |
| REVIEWING | decision=block | BLOCKED(review_block) | — |
| REVIEWING | run 失败 ∨ review.json 修复重试后仍无效 | BLOCKED(review_failed) | — |
| READY_FOR_REVISION | scheduler:slot | REVISING | revision+=1；生成 input.md（含 history digest）；spawn developer |
| WAITING_FOR_HUMAN_APPROVAL | human:approve（SHA+diff_sha256 均匹配） | APPROVED | 写 approvals |
| WAITING_FOR_HUMAN_APPROVAL | human:reject | READY_FOR_REVISION* | reject 理由进入下轮输入；写 approvals(action=reject) |
| APPROVED | human:merge | MERGING | 校验前置条件（§9.5） |
| APPROVED / MERGE_CONFLICT | human:mark_merged_external | MERGED | 清理 worktree（保留分支） |
| MERGING | merge 成功 | MERGED | 清理 worktree；可选删分支 |
| MERGING | 冲突 | MERGE_CONFLICT | `git merge --abort` |
| MERGE_CONFLICT | human:retry_merge | MERGING | — |
| BLOCKED | human:resume_with_guidance | READY_FOR_REVISION（revision=0 时为 READY_FOR_DEVELOPMENT） | guidance 进入下轮输入 |
| BLOCKED | human:force_approve | WAITING_FOR_HUMAN_APPROVAL | 需存在 revision commit |
| 任意非终态 | human:cancel | CANCELLED | kill 活动 run；移除 worktree；保留分支 |

`*`：凡进入 READY_FOR_REVISION，先检查 `revision >= max_revisions`（默认 3，任务级可配）——是则改为 `BLOCKED(max_revisions)`。这是 Claude↔Codex 环路的硬上限。

### 5.2 事务与并发规则

- 每次状态迁移 = 一个 SQLite 事务：`UPDATE tasks.status` + `INSERT events` 必须同事务提交；迁移前校验"当前状态 ∈ 允许来源"，否则返回 `TASK_INVALID_STATE`。
- 非法迁移一律拒绝（含 UI 双击、重复回调）。
- scheduler：全局信号量限制并发 agent run 数（默认 2，可配）；每个任务同一时刻最多 1 个活动 run。
- Agent 永远不直接改状态；状态只因"Orchestrator 验证过的事件"而变。

## 6. 数据模型（SQLite，sqlx migrate 从第一个迁移文件开始）

连接策略：1 个写连接 + 读连接池；`PRAGMA journal_mode=WAL; PRAGMA foreign_keys=ON; PRAGMA busy_timeout=5000;`。所有时间戳 TEXT，ISO-8601 UTC。ID 用 UUIDv7（字符串），任务另有项目内自增 `seq`（显示为 `TASK-007`）。

```sql
CREATE TABLE projects (
  id TEXT PRIMARY KEY,
  seq INTEGER NOT NULL UNIQUE,
  name TEXT NOT NULL,
  repo_path TEXT NOT NULL UNIQUE,          -- 规范化绝对路径
  default_branch TEXT NOT NULL,
  worktree_root TEXT NOT NULL,
  settings_json TEXT NOT NULL DEFAULT '{}',-- per-project 本地设置（含危险开关）
  created_at TEXT NOT NULL, updated_at TEXT NOT NULL
);

CREATE TABLE tasks (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL REFERENCES projects(id),
  seq INTEGER NOT NULL,
  title TEXT NOT NULL,
  description TEXT NOT NULL,
  status TEXT NOT NULL,
  blocked_reason TEXT,
  blocked_detail TEXT,
  developer_agent TEXT NOT NULL,           -- 'claude_code' | 'codex'
  reviewer_agent TEXT NOT NULL,
  target_branch TEXT NOT NULL,
  base_commit TEXT,
  branch TEXT,
  worktree_path TEXT,
  current_revision INTEGER NOT NULL DEFAULT 0,
  max_revisions INTEGER NOT NULL DEFAULT 3,
  created_at TEXT NOT NULL, updated_at TEXT NOT NULL,
  UNIQUE(project_id, seq)
);

CREATE TABLE task_revisions (
  id TEXT PRIMARY KEY,
  task_id TEXT NOT NULL REFERENCES tasks(id),
  revision INTEGER NOT NULL,
  commit_sha TEXT,
  diff_stat_json TEXT,                     -- {files, insertions, deletions, flagged:[...]}
  developer_run_id TEXT,
  created_at TEXT NOT NULL,
  UNIQUE(task_id, revision)
);

CREATE TABLE agent_runs (
  id TEXT PRIMARY KEY,
  task_id TEXT NOT NULL REFERENCES tasks(id),
  revision INTEGER NOT NULL,
  role TEXT NOT NULL,                      -- 'developer' | 'reviewer' | 'validator'
  agent TEXT,                              -- validator 为 NULL
  status TEXT NOT NULL,
  child_pid INTEGER,
  child_started_at TEXT,                   -- 与 PID 一起用于防 PID 复用
  exit_code INTEGER,
  session_id TEXT,                         -- claude --resume 备用
  cost_usd REAL, tokens_in INTEGER, tokens_out INTEGER,
  run_dir TEXT NOT NULL,
  timeout_secs INTEGER NOT NULL, idle_timeout_secs INTEGER NOT NULL,
  started_at TEXT, finished_at TEXT,
  created_at TEXT NOT NULL
);
CREATE INDEX idx_runs_task ON agent_runs(task_id, created_at);

CREATE TABLE reviews (
  id TEXT PRIMARY KEY,
  task_id TEXT NOT NULL REFERENCES tasks(id),
  revision INTEGER NOT NULL,
  run_id TEXT NOT NULL REFERENCES agent_runs(id),
  commit_sha TEXT NOT NULL,
  decision TEXT NOT NULL,                  -- pass/request_changes/block
  summary TEXT,
  raw_path TEXT NOT NULL,
  created_at TEXT NOT NULL
);

CREATE TABLE review_issues (
  id TEXT PRIMARY KEY,
  review_id TEXT NOT NULL REFERENCES reviews(id),
  severity TEXT NOT NULL,                  -- critical/high/medium/low
  file TEXT, line_start INTEGER, line_end INTEGER,
  title TEXT NOT NULL,
  description TEXT, suggested_action TEXT,
  resolved INTEGER NOT NULL DEFAULT 0
);

CREATE TABLE approvals (
  id TEXT PRIMARY KEY,
  task_id TEXT NOT NULL REFERENCES tasks(id),
  revision INTEGER NOT NULL,
  commit_sha TEXT NOT NULL,
  diff_sha256 TEXT NOT NULL,
  action TEXT NOT NULL,                    -- approve/reject
  reason TEXT,
  created_at TEXT NOT NULL
);

CREATE TABLE events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,    -- 全局单调序
  task_id TEXT, run_id TEXT, revision INTEGER,
  actor TEXT NOT NULL,                     -- orchestrator/agent/human/system
  event_type TEXT NOT NULL,
  payload_json TEXT NOT NULL,
  created_at TEXT NOT NULL
);
CREATE INDEX idx_events_task ON events(task_id, id);

CREATE TABLE artifacts (
  id TEXT PRIMARY KEY,
  task_id TEXT NOT NULL, revision INTEGER,
  kind TEXT NOT NULL,                      -- patch/test_report/review_raw/handoff/export
  path TEXT NOT NULL, sha256 TEXT,
  created_at TEXT NOT NULL
);

CREATE TABLE settings (key TEXT PRIMARY KEY, value_json TEXT NOT NULL);
```

`events.jsonl` 导出：`events_export` 命令把某项目全部事件写到 `<app_data>/exports/`，仅此用途，不参与任何决策。

## 7. 进程监督（process-supervisor crate）

- 使用 `tokio::process` + `command-group` crate：Unix 建进程组、Windows 建 Job Object；取消/超时时**整组击杀**（Claude Code / Codex 会派生 shell、node、MCP 子进程，只杀主进程必留孤儿）。
- 杀进程流程：SIGTERM(组) → 宽限 10s → SIGKILL(组)；Windows 直接 TerminateJobObject。
- 记录 `child_pid` + `child_started_at`；恢复流程判定"进程仍存活"必须同时匹配 PID 与启动时间（读 /proc 或 sysinfo crate），防 PID 复用误杀无辜进程。
- 双超时：绝对超时（developer 默认 1800s、reviewer 900s、validator 每步按 project.toml）+ 空闲 watchdog（默认 300s 无任何 stdout/stderr 字节即判卡死→按 TIMED_OUT 处理）。
- 输出捕获：stdout/stderr 流式写入 run 目录文件；单文件上限 64MB，超限截断并记事件；转发 UI 的日志块做批量化（≤10 批/秒/run，每批 ≤64KB）与脱敏（见 §11）。
- 环境变量：继承用户环境（CLI 需要各自的登录态），支持 per-project denylist 剔除指定变量；`LC_ALL=C` 仅注入给 git 子进程调用，不注给 Agent。
- 工作目录一律传规范化绝对路径；Windows 上先做路径规范化（去 `\\?\` 歧义、统一分隔符），并必须通过含中文与空格路径的测试（论文 M233 教训）。

## 8. Agent 接入（agent-adapters crate）

### 8.1 Adapter 接口（v1.1）

```rust
#[async_trait]
pub trait AgentAdapter: Send + Sync {
    fn kind(&self) -> AgentKind;
    async fn detect(&self, env: &CliEnv) -> Result<AgentInstallation>; // 路径+版本+能力探测
    fn capabilities(&self) -> AgentCapabilities;  // { streams_events, native_output_schema, supports_resume, read_only_mode }
    async fn start(&self, req: AgentRunRequest, tx: EventSender) -> Result<RunningAgent>;
    async fn cancel(&self, run: &RunningAgent, reason: CancelReason) -> Result<()>;
    async fn collect_result(&self, run_dir: &Path, role: RunRole) -> Result<CollectedResult>;
}
```

`AgentRunRequest` 包含：worktree 路径、role、input 文件相对路径、超时、权限档位（normal / yolo）、追加白名单命令（来自 project.toml）。适配器把原始输出流归一化为 `AgentEvent`（见 03 §4）写入 `agent-events.jsonl` 并转发 UI。

### 8.2 Claude Code 适配器

开发角色（在 worktree 内执行）：

```bash
claude -p "请完整阅读 .agentflow-in/input.md，按其要求完成任务；结束前必须按其中的说明写出 .agentflow-out/result.json" \
  --output-format stream-json --verbose \
  --permission-mode acceptEdits \
  --allowedTools "Edit,Write,Read,Grep,Glob,Bash(git status:*),Bash(git diff:*),Bash(git log:*)<,项目白名单命令如 Bash(bun test:*)>" \
  --max-turns 100
```

要点：
- **必须用 `stream-json`（需 `--verbose`）**：增量 JSONL 事件既是空闲 watchdog 的存活信号，也提供最终 `type:"result"` 事件（含 `is_error`、`result`、`session_id`、`total_cost_usd`、usage）。`--output-format json` 只在结束时输出、会被 watchdog 误杀，禁止使用。
- `acceptEdits` 自动放行文件编辑与常见文件系统命令；其余 shell 命令必须在 `--allowedTools` 白名单内（形如 `Bash(bun test:*)`），白名单来自 project.toml 的 validate 步骤 + 项目配置的额外允许命令。未在白名单内的命令被 CLI 拒绝——这就是安全边界的实际执行点。
- **禁止 `--bare`**（它跳过 OAuth/keychain，订阅登录用户会直接失败）。
- `--dangerously-skip-permissions` 仅在 per-project 本地设置里显式打开"完全放权模式"时使用，UI 有醒目警告；默认关闭。
- 审查角色（当用户把角色互换时）：`--allowedTools "Read,Grep,Glob,Bash(git diff:*),Bash(git log:*)" --disallowedTools "Write,Edit"`，最终消息即 review JSON，从 result 事件的 `result` 字段提取（先剥 ```json 围栏）→ schemars 校验 → 失败走修复重试（§10）。
- `session_id` 入库；返工轮可选 `--resume <session_id>`（默认不用，作为设置项，保持每轮干净上下文的默认原则）。
- 版本/能力探测：`claude --version`；对 `--help` 输出 grep 关键 flag（`--output-format`、`--permission-mode`），缺失则报 `ENV_CLI_INCOMPATIBLE`。**所有 flag 只允许出现在适配器 crate 内**，业务代码零散落。

### 8.3 Codex 适配器

审查角色（默认配置，只读沙箱 + 原生 schema 校验）：

```bash
codex exec --cd <worktree> --sandbox read-only \
  --output-schema <schemas/review.schema.json> \
  -o <run_dir>/last-message.json \
  --json \
  "请完整阅读 .agentflow-in/review-input.md，独立审查其中指定的 revision commit，并按 schema 输出审查 JSON"
```

- `--json` 的 JSONL 事件流做存活信号与日志；最终消息由 `-o/--output-last-message` 落盘到 run 目录（app data 内，Codex 沙箱外由 CLI 自身写入，不受 read-only 限制）。
- `--output-schema` 让 Codex 原生校验输出形状；Orchestrator 仍二次校验（不信任单点）。
- 只读沙箱 = 审查者物理上改不了代码，独立性由沙箱保证而非提示词。

开发角色（角色互换时）：`codex exec --cd <worktree> --sandbox workspace-write --json -o ...`，成果写 `.agentflow-out/result.json`（workspace-write 可写工作区）。`codex exec` 永不弹确认（无 TTY，approval 自动为 never），因此沙箱档位就是唯一安全旋钮——**禁止 `danger-full-access` / `--yolo`**，除非用户打开完全放权模式。

### 8.4 Gemini CLI / Qwen Code 适配器

- Gemini CLI：开发使用无头 `stream-json`、AgentFlow worktree 内的 sandbox + yolo；审查使用 `json` 信封并强制 `--approval-mode plan` 只读模式，Orchestrator 从 response 字段取出并二次校验 review JSON。
- Qwen Code：开发使用 `stream-json`、sandbox 和单 run 时间/工具调用预算；审查使用 `--approval-mode plan` + `--json-schema @review.schema.json`，stdout 直接收取已验证 JSON。
- 两者都只提供适配、能力探测和路径配置；首次阶段不替用户下载安装。缺失时 onboarding 明确显示“适配器已就绪、CLI 未安装”。

### 8.5 CLI 发现（env-check / onboarding-check）

顺序：用户显式配置的绝对路径 → PATH 查找 → 补救探测。补救探测：
- macOS/Linux：GUI 启动的应用只有 launchd 的贫瘠 PATH，找不到 homebrew/nvm/bun 下的 `claude`/`codex`/`gemini`/`qwen`。用 `$SHELL -lic 'command -v <tool>'`（登录 shell）探测一次并缓存结果；失败则 UI 引导用户手填路径。
- Windows：`where.exe`；探测结果统一规范化；长路径注册表状态检测并提示。
- 探测产物 `EnvReport`：每个 CLI 的 path/version/兼容结论、可安全查询时的登录状态 + git 版本。`OnboardingReport` 另含 daemon 在线状态、推荐角色组合、阻塞提示与存储占用；Gemini/Qwen 没有稳定的无副作用登录查询命令，凭据在首次 headless run 验证。

## 9. 工作流各阶段实现细节

### 9.1 任务启动

1. `base_commit = git rev-parse <target_branch>`；
2. `git worktree add <wt_path> -b agentflow/TASK-<seq> <base_commit>`；
3. 追加 info/exclude（幂等）；
4. 生成 `.agentflow-in/input.md`（模板见 03 §6.2：任务描述 + 内联 rules + 输出约定 + 禁改清单声明）。

### 9.2 验证（VALIDATING）

- 逐条执行 `project.toml [[validate.steps]]`：`name` + `argv`（数组，直接 spawn，**绝不经过 shell**）+ `timeout_secs`，cwd=worktree；
- 产出 `r<N>-tests.json`：`{passed, steps:[{name, argv, exit_code, duration_ms, stdout_tail, stderr_tail}]}`（tail 各 8KB）；
- spawn 失败/命令不存在 = 基础设施错误 → BLOCKED(validation_infra)，与"测试失败"严格区分。

### 9.3 审查输入生成（独立性规则）

`review-input.md` 只包含：原始需求（title+description+人工 guidance 历史）、revision commit SHA、精简 diff、测试报告、审查输出约定。**不包含开发者的 result.json summary**（防锚定；summary 只给 UI 展示给人看）。

精简 diff 规则：
- `git diff <base>..<sha> --find-renames --no-color`，按 project.toml `review.exclude_globs`（默认含各类 lockfile、`dist/**`、`*.min.*`）过滤；二进制文件（numstat 为 `-`）只列名不给内容；
- 总量超 `review.max_patch_bytes`（默认 256KB）→ 降级为"numstat 文件清单 + 提示审查者在 worktree 内自行 `git show <sha>` / 读文件"（审查者本就在代码现场，diff 文件只是便利品）；
- **凡 diff 触及 `.agentflow/`、`CLAUDE.md`、`AGENTS.md`、`.claude/`、CI 配置**：写入 `diff_stat_json.flagged`，审查输入中显式标注"以下文件属于控制面/规则文件，请重点审查其变更是否合理"，UI 端同样高亮（见 02 文档）。

### 9.4 返工输入（handoff digest）

REVISING 的 input.md 额外内联 `history.md`：每一轮的（revision 号、commit SHA、开发自述 summary——给同一开发者自己看不算锚定、审查 decision、未 resolved 的 issues、测试失败摘要、人工 guidance/reject 理由），总量截断 8KB（保留最近轮次优先）。

### 9.5 审批与合并

- 审批请求携带 `(revision, commit_sha, diff_sha256)`；服务端重算 diff_sha256（对 §9.3 之前的**完整**未过滤 patch 做 SHA-256），不匹配返回 `DIFF_STALE`。
- 合并前置条件（不满足返回 `MERGE_PRECONDITION_FAILED` 及原因）：主检出在 `target_branch` 上、`git status --porcelain -z` 为空、target HEAD 从 base 之后的推进不影响 merge 可行性（直接尝试）；执行 `git merge --no-ff agentflow/TASK-<seq> -m "[agentflow] merge TASK-<seq>: <title>"`；冲突则 `git merge --abort` → MERGE_CONFLICT。
- 用户也可选择"我自己合并"→ `mark_merged_external`。
- revision commit 规范：message `[agentflow] TASK-<seq> r<N>: <title>`，author `AgentFlow (<agent>) <agent@agentflow.local>`。

## 10. 结构化输出与"文件先于信号"

- 权威成果 = 文件：开发 run 是 `.agentflow-out/result.json`，审查 run 是 `last-message.json`。进程退出码、stream 事件、（M4 的）MCP 调用统统只是"该去读文件了"的提示。收到任何完成信号后：回读文件 → schemars 校验（03 §6 的 schema，含 `schema_version`）→ 才允许状态迁移。
- input.md 中要求 Agent **先写临时文件再 rename** 到 result.json（原子写）；Orchestrator 读取时若 JSON 截断按无效处理。
- 修复重试环：文件缺失或校验失败 → 以同一 session（Claude 用 `--resume`，Codex 用 `codex exec resume`；不支持时带上下文重新 spawn）追加一条"输出无效，错误如下：<validator errors>，请只重新输出合法 JSON"，**仅重试 1 次**；再失败 → BLOCKED。
- `changed_files` 等自述字段仅作参考：实际变更以 `git diff` 为准，两者不一致记 `result_claim_mismatch` warning 事件。

## 11. 安全边界 v1.1

1. **执行主体**：模型生成命令的允许/拒绝由 Claude Code 权限系统（allowedTools/permission-mode）与 Codex 沙箱（--sandbox）执行；Orchestrator 的职责是（a）以最小权限档位配置它们（b）把所用配置完整记入 run 事件以供审计。
2. Orchestrator 自身执行的命令（git、validate 步骤）一律 argv 数组直接 spawn，无 shell 展开。
3. Agent 可写面 = 自己的 worktree（+ Codex 的 /tmp）；应用数据目录、其他项目、主检出对 Agent 的写路径均不授予。
4. 控制面文件的变更（§9.3 flagged 清单）永远显式呈现给审查者与人类。
5. 审查结论是建议不是决定：无论 review 说什么，进入 WAITING_FOR_HUMAN_APPROVAL 后人类看到的是 diff 本身；恶意 diff 诱导审查者放行的风险由"人工批准 + flagged 高亮"兜底。
6. 日志脱敏：转发 UI 与导出的文本过一遍规则集（AKIA…、ghp_…、sk-…、`Authorization: Bearer`、`password=` 等模式打码）；run 目录内原始日志不改写，但只对 MERGED/CANCELLED 任务默认保留 14 天。
7. 完全放权模式（`--dangerously-skip-permissions` / `danger-full-access`）：per-project 本地设置，默认关，UI 需二次确认并常驻警示条。
8. 合并、删分支、清 worktree 均需人工动作触发；Orchestrator 永不自动 push。

### 11.1 存储生命周期与删除语义

- `agentflowd` 启动后立即执行一次维护，之后每 6 小时执行；默认原始 run 日志 14 天、缓存 2 GiB LRU、回收站 7 天。
- 正式项目文件、revision commit、patch、Review 摘要和测试摘要不属于自动清理对象。
- 用户可选择 `logs`（只删 stdout/stderr/事件/模型原文）、`runtime`（另清 AgentFlow worktree）或 `everything`（整任务入回收站）。仍可能继续审批/合并的任务禁止 runtime/everything 清理。
- `everything` 先原子移动任务目录并把任务标记为 deleted；列表立即隐藏，但可 restore。到期或“清空回收站”后才按外键顺序删除 SQLite 记录与文件。
- 第三方 CLI 的全局会话目录不由 AgentFlow 静默删除；AgentFlow 只删除自己的 app-data 与专属 worktree。

## 12. 恢复与崩溃处理

启动时恢复扫描（顺序执行，全部记事件）：
1. `SELECT * FROM agent_runs WHERE status='RUNNING'` → 对每条：按 PID+启动时间判定进程是否仍存活 → 存活则整组击杀 → run 置 `INTERRUPTED`；
2. 所属任务回滚到该 run 之前的可运行状态：developer run → revision==1 ? READY_FOR_DEVELOPMENT : READY_FOR_REVISION（并 revision-=1，因为该轮未产出 commit）；validator → VALIDATING；reviewer → READY_FOR_REVIEW。**不自动重跑**，等 scheduler/用户触发（确定性优先）；
3. worktree 一致性：任务记录了 worktree_path 但目录缺失/损坏 → BLOCKED(worktree_missing)，提供"从分支重建 worktree"修复动作；
4. 半成品输入目录（`.agentflow-in/`）在下次 run 前整目录重建，天然幂等。

由 §5.2 的事务规则保证：不存在"状态改了但事件没记"或反之的中间态。

## 13. 本地 MCP（M4，可选增强，不改变控制权）

- 形态：独立 stdio 二进制 `agentflow-mcp`，注册进 worktree 层面的 Claude `.mcp.json` / Codex `-c mcp_servers.*` 覆盖；它经 Unix domain socket / Windows 命名管道连回 Orchestrator。零端口、零 token、零 Origin 校验需求。
- 工具集：`get_task` / `report_progress` / `submit_work` / `get_review_task` / `submit_review`。其中 submit_* 依然只是信号：Orchestrator 收到后照旧回读文件校验（§10 不变式对 MCP 同样成立）。
- MCP 不负责启动 Agent、不承载状态机、失败不影响文件路径闭环。

## 17. 运行治理 v1（2026-07-20）

本阶段在既有 revision commit 与事件审计之上增加六条纵向能力：

1. **可复现运行与质量评估**：每轮验证保存 commit、输入/patch/验证配置哈希、平台和工具版本；重放只在固定 commit 上执行相同 argv。质量分由验证、审查、控制面文件和未关闭高风险问题等确定性检查组成，不让模型自行打分。
2. **编码前计划审批**：任务可启用计划门禁。开发 Provider 先以只读权限输出 `plan.json`，状态进入 `WAITING_FOR_PLAN_APPROVAL`；人工批准后才进入开发，驳回理由进入下一版计划输入。
3. **Token、费用与时间预算**：预算在任务创建时固化。每次 run 前检查累计值并把剩余时间收紧为本次 timeout；run 后再次检查。Claude/Codex/API 用量统一归一化，API 费用使用供应商返回值或用户保存的项目级价格快照，绝不内置可能过期的价格。超限进入 `BLOCKED(budget_exceeded)`，不会自动切换 Provider 绕过限制；人工提高/移除上限后从保存的安全检查点恢复。
4. **合并撤销与回滚**：合并前后 commit 都入库。`undo` 仅在主分支干净且 HEAD 仍等于本任务 merge commit 时允许；若已有后续提交只能使用 `revert` 创建新提交，禁止隐式改写共享历史。
5. **GitHub/GitLab PR 与 CI**：交付模式是任务策略而不是全局猜测。通过 `gh`/`glab` 的 argv 调用创建或复用 PR/MR，保存 URL/编号/CI 状态；CI 未通过时不进入已合并态。
6. **远程执行节点**：节点只保存 SSH 地址、用户、端口和工作根目录，不保存私钥/口令。健康检查使用 BatchMode。v1 远程范围为固定 revision 的验证与质量重放：以 `git archive` 发送不可变提交，逐参数安全引用验证命令，把报告和环境指纹留在本地后清理远端临时目录。开发/审查仍保留本地 Provider 沙箱，后续可在相同节点协议上扩展。
